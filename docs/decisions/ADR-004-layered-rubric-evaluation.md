# ADR-004：采用分层 Rubric 评分架构作为目标演进方向

## 状态

已接受。

## 背景

当前仓库已经明确：

- 系统核心能力是 `LLM as Judge`
- Prompt 只允许在后端治理
- 正式输出必须是严格 JSON
- 当前仓库处于结构与治理优先阶段

与此同时，项目的长期目标并不只是复制一次性评分体验，而是建立一套可演进、可回归、可校准的小说评价机制。

如果长期继续依赖“单次强模型整体直评”，会出现以下问题：

- 评分过程解释性不足
- Prompt 和 Provider 变化时难以定位漂移来源
- 低成本模型较难承担局部任务
- 难以对评分中的证据、冲突和不可评情况做显式治理

## 决策选项

### 方案 A：继续以单次大模型整体直评为主

优点：

- 实现最简单
- 运行时链路最短
- 当前阶段最容易快速出结果

缺点：

- 难解释
- 难校准
- 难以拆分成本
- 一旦结果波动，难以定位问题源头

### 方案 B：完全分层的小模型流水线

优点：

- 最强的可解释性
- 最适合阶段契约治理
- 最适合做局部评测和版本回归

缺点：

- 早期实现成本高
- 流程复杂，容易出现级联错误
- 在仓库当前阶段过早引入运行时复杂度

### 方案 C：分层 Rubric + 小模型优先 + 升级仲裁

优点：

- 能把大部分结构化子任务交给低成本模型
- 能在争议样本上保留更强模型的稳定性收益
- 更适合渐进落地
- 兼顾可解释性、成本和治理能力

缺点：

- 比单次整体直评更复杂
- 需要额外治理阶段契约、路由阈值和仲裁逻辑

## 决策

采用 **方案 C：分层 `rubric` + 小模型优先 + 升级仲裁** 作为正式评分主线。

当前阶段的落地策略是：

- 以 `docs/architecture/layered-rubric-evaluation-architecture.md`、`docs/contracts/rubric-stage-contracts.md`、`docs/planning/layered-rubric-implementation-plan.md` 作为正式实施依据
- Prompt、Schema、Evals 与运行时编排应全部围绕同一套分层 `rubric` 主线推进

采用该主线后，按以下策略推进：

- 当前阶段先建立与该 `rubric` 对齐的基线评分路径
- 在基线稳定后，再逐步拆出证据抽取、原子评分、一致性校验和升级仲裁
- 对外正式输出结构保持稳定，不因内部评分机制演进而频繁变动

## 具体原则

### 1. 内部判断细化，外部结构稳定

内部可以拥有比对外字段更细的 `rubric` 维度，但对外仍以统一正式结果对象作为唯一业务输出。

### 2. 证据优先

所有核心评分都应能够追溯到证据，而不是只保留最终结论。

### 3. 小模型承担大部分结构化任务

低成本模型优先用于：

- 输入预检查
- 证据抽取
- 原子评分
- 一致性复核

### 4. 强模型只处理争议和高价值样本

更强模型默认不参与全部样本，而是只在低置信度、高冲突、高价值或高风险样本中触发仲裁。

### 5. Prompt、Schema 与 Evals 必须联动治理

- Prompt 变化应回归
- Provider 变化应回归
- Schema 变化应回归
- 阶段契约变化应纳入评测范围

## 预期结果

若该提案被接受，系统将形成以下长期收益：

- 降低对单个高成本模型的强依赖
- 提高评分结果的可解释性和可审计性
- 更容易比较不同 Provider 与不同 Prompt 版本
- 为后续引入人工复核或保守路由策略留出接口

同时，也会引入以下治理成本：

- 需要维护更多 Prompt 资产
- 需要维护阶段契约与路由规则
- 需要在 `evals/` 中增加阶段级评测与校准分析

## 后续动作

- 先完成分层 `rubric` 的架构文档与契约文档
- 在评分流水线、JSON 契约、评测框架中补入该机制
- 先建立基线路径，再逐步演进到完整分层路径
