# ADR-004：采用全 LLM 分阶段 Rubric 评分架构作为正式主线

## 状态

已接受。

## 背景

当前仓库已经明确以下长期约束：

- 系统核心能力是 `LLM as Judge`
- Prompt 只允许在后端治理
- 正式输出必须是严格 JSON
- 当前仓库仍处于结构与治理优先阶段

当前需求已经明确固定新的正式评分方式：先由 `LLM` 按 `rubric` 进行分点评价，再由新的聚合模型汇总这些评价结果并输出最终结果。

因此，仓库需要直接围绕这条单主线冻结文档、契约与后续实现边界，而不是为其它评分路径保留兼容口径。

## 决策选项

### 方案 A：继续以单次大模型整体直评为主

优点：

- 实现最简单
- 运行时链路最短
- 当前阶段最容易快速出结果

缺点：

- 评分过程解释性不足
- Prompt 和 Provider 变化时难以定位漂移来源
- 难以把分点评价沉淀为可回归的正式资产
- 容易重新回到黑盒整体判断

### 方案 B：全 LLM 分阶段 Rubric 评分 + 聚合模型输出

优点：

- 能先冻结 `rubric` 分点评价，再由聚合模型统一收口
- 兼顾结构化解释与最终结果表达
- 更适合围绕 Prompt、Schema 与 Evals 做版本治理
- 形式上只有一条正式主线，文档与实现边界更干净

缺点：

- 比单次整体直评更复杂
- 多阶段模型调用会提高时延与成本
- 若阶段契约设计不清晰，聚合模型仍可能黑盒化

### 方案 C：保留额外分叉评分路径

优点：

- 理论上具备更细的成本分层空间
- 理论上可以为边界样本增加额外处理层

缺点：

- 与当前需求冲突
- 会继续保留并行路径与额外治理负担
- 会让正式主线长期处于迁移中状态，不利于当前仓库收口

## 决策

采用 **方案 B：全 `LLM` 分阶段 `rubric` 评分 + 聚合模型输出** 作为新的正式评分主线。

正式主线固定为五段：

1. `输入预检查`
2. `LLM rubric 分点评价`
3. `轻量一致性整理`
4. `新模型聚合输出`
5. `正式结果投影`

该主线要求：

- 只保留一条正式评分路径
- 不引入额外成本分层原则作为正式主线前提
- 不设置额外分叉升级阶段
- 不预留外部介入型正式入口或结构
- 聚合模型必须以分点评价结果作为正式输入边界，不直接替代分点评价阶段

## 具体原则

### 1. 内部判断分层，外部结构稳定

内部允许存在比分顶层结果更细的 `rubric` 维度和阶段对象，但对外仍以统一正式结果对象作为唯一业务输出。

### 2. 分点评价先于最终结论

最终顶层分数、强弱项、市场判断和编辑结论，都应建立在前序分点评价结果之上，而不是直接由聚合模型绕过中间阶段生成。

### 3. 证据嵌入分点评价项

不再将“证据抽取”作为长期独立主阶段，而是将证据引用直接嵌入每个分点评价项中，使评分理由与文本依据保持绑定关系。

### 4. 一致性能力保留，但不再承担路由职责

一致性能力用于发现缺项、冲突、重复处罚和无依据判断，但不再驱动额外分叉流程，也不再作为外部介入入口。

### 5. Prompt、Schema 与 Evals 必须联动治理

- Prompt 变化应回归
- Provider / Model 变化应回归
- Schema 变化应回归
- 阶段契约变化应纳入评测范围

## 预期结果

若该提案被接受，系统将形成以下长期收益：

- 正式主线更单一，文档与实现边界更清晰
- 分点评价结果可用于解释、回归和质量分析
- 聚合模型可以在不破坏对外契约的前提下稳定输出最终结果
- Prompt、Schema 与 Evals 能围绕同一条主线同步演进

同时，也会引入以下治理成本：

- 需要维护更多正式阶段对象与 Prompt 资产
- 需要专门评估分点评价稳定性与聚合输出稳定性
- 需要在 `evals/` 中记录更多阶段级结果差异
