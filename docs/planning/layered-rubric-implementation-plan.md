# 分层 Rubric 实施计划

## 目标

本文档将“分层 `rubric` 评分机制”拆解为可执行的文档、契约、Prompt、Schema、Evals 与运行时建设步骤，用于指导仓库从当前结构建设阶段逐步演进到可治理的多阶段评分系统。

## 当前前提

当前仓库已明确以下约束：

- Prompt 只能在后端治理
- 正式输出必须是严格 JSON
- `packages/schemas/` 是正式结构的唯一真源
- `evals/` 用于评测样本、用例、基线与报告
- 当前仓库仍处于文档与结构优先阶段

因此，本实施计划遵循：

- 先文档和契约
- 再评测和版本治理
- 最后再进入具体运行时实现

## 总体策略

推荐采用“分层 `rubric` 单主线、分阶段落地”的策略：

- **正式主线**：分层 `rubric` + 小模型优先 + 升级仲裁
- **当前落地起点**：先冻结词表、契约和评测标准，再建立与 `rubric` 对齐的基线评分路径

这样做的原因：

- 可以先冻结词表、契约和评测标准
- 可以避免在仓库早期同时引入过多运行时复杂度
- 可以让后续 Prompt、Schema、Evals 与运行时围绕同一主线演进

## 分阶段路线

### Phase 0：Rubric 机制边界冻结

目标：

- 冻结 `rubric` 术语与分层职责
- 统一输入类型边界
- 统一内部维度与外部字段映射口径
- 明确分层 `rubric` 是唯一正式主线

交付物：

- 本文档
- 架构文档
- 契约文档
- ADR 文档

完成标准：

- 团队能使用统一词表讨论评分机制
- 不再把 `rubric`、Prompt、输出字段混为同一层概念
- 文档层已明确只有一条正式主线

### Phase 1：建立 Rubric 对齐基线

目标：

- 保持现有对外结果字段稳定
- 建立一套与新 `rubric` 对齐的基线评分链路
- 让后续任何评分方案都能回到同一评测基线比较

建议工作：

- 冻结一级维度与二级子维度词表
- 冻结原子评分锚点
- 定义顶层字段的来源关系与上限约束
- 为基线评分路径补充结构化解释要求

完成标准：

- 顶层四个分数与内部一级维度关系明确
- 至少一版评分 Prompt 能围绕新 `rubric` 产出稳定结构
- Evals 能开始记录基线分布

### Phase 2：拆出证据抽取与原子评分

目标：

- 将“证据抽取”和“原子评分”从整体评分中拆分出来
- 评估低成本模型在结构化子任务上的稳定性

建议工作：

- 为 `InputScreeningResult`、`EvidencePack`、`RubricScoreCard` 设计正式 Schema
- 新增证据抽取 Prompt 与原子评分 Prompt
- 记录证据覆盖率与原子分数稳定性

完成标准：

- 原子评分能绑定证据引用
- 阶段契约合法率达到可接受水平
- 与基线路径比较时，结构稳定性不下降

### Phase 3：增加一致性校验与校准

目标：

- 识别维度冲突、证据不足和重复计分
- 将顶层分数逻辑校准从隐性 Prompt 能力转为显性流程节点

建议工作：

- 定义 `ConsistencyCheckResult` 与 `AggregatedRubricResult`
- 为一致性校验和校准建立 Prompt 与规则结合机制
- 将分数漂移、冲突率、证据不足率接入 Evals

完成标准：

- 能识别关键冲突并给出结构化说明
- `signingProbability` 等顶层字段的逻辑关系更加稳定
- 与基线路径比较时，解释性和一致性有明确提升

### Phase 4：接入升级仲裁

目标：

- 只在必要时调用更强模型
- 用更强模型解决争议项，而不是替代全部流程

建议工作：

- 定义升级阈值
- 定义 `ArbitrationRecord`
- 评估升级命中率、升级成本和升级收益

完成标准：

- 升级触发规则可解释
- 升级样本比例受控
- 升级后的质量提升可在 Evals 中被量化

### Phase 5：形成长期运行治理闭环

目标：

- 将 Prompt、Schema、Evals、Provider 变更统一纳入治理流程

建议工作：

- Prompt 版本与评测基线绑定
- Schema 变更要求回归
- Provider 切换要求回归
- 评估是否引入人工复核或高价值任务保守模式

完成标准：

- 版本变更可以清晰定位影响
- 评分系统不再依赖单个 Prompt 的隐式稳定性

## 推荐样本分层

后续 `evals/` 建议至少覆盖以下样本类型：

- 常规样本：主流网文开篇、普通章节、标准大纲
- 边界样本：题材混搭、视角复杂、实验性表达
- 冲突样本：市场吸引力强但写作执行差，或反之
- 证据不足样本：过短、断裂、信息不足
- 风险样本：明显 AI 味、术语污染、套路拼贴
- 升级样本：低置信度、高冲突、高价值样本

## 关键指标

### 结构指标

- 最终结果 JSON 合法率
- 阶段契约 JSON 合法率
- 缺失字段率

### 质量指标

- 顶层分数稳定性
- 原子评分稳定性
- 证据覆盖率
- 证据与结论一致率
- 不可评识别准确率

### 路由指标

- 升级触发率
- 升级后修正率
- 无效升级率
- 平均时延与平均成本

### 回归指标

- Prompt 变更前后漂移
- Provider 变更前后漂移
- 分层路径相对基线路径的收益

## 风险与约束

### 风险 1：阶段过多，成本失控

应对：

- 默认走低成本路径
- 明确升级阈值
- 不将所有阶段都绑定到强模型

### 风险 2：阶段契约太早冻结，反而限制演进

应对：

- 先冻结对外结果结构
- 内部阶段契约采用版本化演进
- 先固定字段语义，再逐步细化

### 风险 3：原子评分看起来更细，但整体判断反而变差

应对：

- 持续与基线路径对比
- 使用一致性校验与校准层
- 在 Evals 中观察解释性收益是否真正转化为质量收益

### 风险 4：升级仲裁变成隐式兜底，掩盖上游问题

应对：

- 仲裁必须记录触发原因
- 统计升级命中率和修正率
- 不允许默认每次都升级

## 验收标准

- 已有文档能说明“为什么使用分层 `rubric` 机制”
- 已有文档能说明每一层的职责、输入、输出、失败模式和升级条件
- 已有文档能说明该机制如何映射到 Prompt、Schema、Evals 与后续实现目录
- 已有文档能解释为什么该机制适合“小模型优先、必要时升级”的路线
- 已有文档不破坏当前对外结果契约和 Prompt 治理原则
