# Phase 1 MVP 范围说明

## 文档目的

本文档用于冻结小说智能打分系统 `Phase 1` 的最小可开发范围，为后端接口设计、应用层编排、Schema 落位、Prompt 资产治理和前端联调提供统一边界。

本文档回答的问题是：

- `Phase 1` 到底做什么
- `Phase 1` 明确不做什么
- 哪些能力属于当前开发范围
- 哪些能力应被推迟到后续阶段
- 哪些交付物完成后，可以按文档进入正式开发

本文档不负责：

- 替代正式 API 契约
- 替代正式 Schema 定义
- 替代 Prompt 正文设计
- 替代后续实现细节

## 适用范围

本文档适用于：

- `apps/api/`
- `packages/application/`
- `packages/domain/`
- `packages/provider-adapters/`
- `packages/schemas/`
- `prompts/`
- `evals/`

本文档也为前端的 Mock / Adapter 对接提供后端实现范围边界，但不重新定义前端信息架构。

## Phase 1 总目标

`Phase 1` 的目标不是一次性实现完整运行时，而是先建立一条**可运行、可治理、可回归的正式评分主线**。

这条主线需要满足：

- 能接收作者提交的联合投稿包并创建评测任务
- 能以统一任务语义返回任务状态
- 能生成符合正式 JSON 结果结构的评测结果
- 能为后续 Prompt、Schema、Provider 和 Evals 演进保留治理接口
- 能支持前端围绕正式结果对象与任务状态进行联调

## Phase 1 产出定位

`Phase 1` 产出的是：

- 一套可实现的 API v0 契约
- 一套最小任务状态与错误语义
- 一条正式评分流程说明与实现边界
- 一版正式输出结构语义与真源位置冻结
- 一套可治理的 Prompt 生命周期规则
- 一套最小可执行的 Evals 基线说明

`Phase 1` 不追求：

- 多路径评分体系
- 复杂分叉路由与多模型升级机制
- 额外介入链路与后台处置流程
- 多 Provider 生产级切换能力
- 全量运营化和后台化能力

## 本阶段支持的输入范围

### 支持的正式输入对象

`Phase 1` 以联合投稿包为正式输入模型，支持：

- `chapters + outline`
- `chapters only`
- `outline only`

说明：

- `chapters + outline` 是正式推荐输入形态
- `chapters only` 与 `outline only` 允许进入降级评估
- `Phase 1` 不再把 `opening`、`chapter`、`outline`、`other` 当作并列正式输入主线
- 任意“其它文本”不进入正式主线一等公民范围

### 支持的输入来源

`Phase 1` 支持：

- 直接文本输入
- 文件上传入口的接口占位与基础业务语义

说明：

- 文件上传在 `Phase 1` 中以“接口边界已明确”为主，不要求先把复杂解析链路做到完备
- 如文件解析能力暂未稳定，可在实现中将其限制为受控能力，但文档上应明确其边界

## 当前技术路线前提

`Phase 1` 当前已冻结以下后端实现基线：

- Web API 框架：`FastAPI`
- 边界 DTO 与运行时校验：`Pydantic`
- 默认且唯一正式 Provider：`DeepSeek API`
- 多阶段 LLM 编排框架：`PocketFlow`
- 部署定位：开源项目、本地部署、本机启动前后端与 worker

说明：

- 本文档负责冻结范围，不替代 `docs/architecture/backend-technical-route.md` 的技术路线真源地位
- 后续实现应围绕该基线展开，而不是再次回到技术中立状态

## 本阶段支持的后端能力

### 1. 任务创建与任务读取

本阶段支持：

- 创建评测任务
- 读取任务详情
- 读取任务结果
- 读取首页摘要数据
- 读取按任务组织的历史记录

### 2. 正式评分链路

本阶段支持一条正式评分链路：

1. 接收联合输入
2. 执行边界内输入校验
3. 构建任务对象
4. 选择 `rubric` / Prompt / Provider / 执行配置
5. 执行输入预检查
6. 执行新 `8` 轴 `LLM rubric` 分点评价
7. 执行轻量一致性整理
8. 由新模型聚合到旧四维骨架层并输出正式结果草案
9. 校验输出是否满足正式 JSON 契约
10. 返回任务状态与结果

说明：

- 本阶段围绕正式评分主线推进
- 本阶段不再为旧路线保留双路径或分叉式扩展口径
- 本阶段不引入 `pairwise`

### 3. 正式结果返回

本阶段支持返回面向前端消费的正式结果对象，包括：

- 顶层四个评分字段
- 平台推荐
- 市场判断
- 编辑结论
- 详细分析
- 优势与弱点
- 基础元信息

说明：

- 结构语义以正式 Schema 为准
- 文档可引用 `docs/contracts/json-contracts.md`，但不以文档替代 Schema 本体

### 4. 最小治理能力

本阶段支持以下治理能力：

- Prompt 版本化管理规则
- Schema 版本治理规则
- Evals 基线运行规则
- Provider 抽象边界约束
- 联合输入与新 `8` 轴词表治理

说明：

- 本阶段强调治理规则先于复杂实现能力
- 复杂运行时能力可后续补齐，但治理边界不能缺席

## 本阶段明确不做的内容

### 不做超出正式主线的额外流程

本阶段不要求实现：

- 额外评分分支
- 多模型升级分流机制
- 额外介入入口
- 后台处置流程
- `pairwise` 评分路径

### 不做多 Provider 生产级对比能力

本阶段不要求：

- 多 Provider 动态路由
- 多 Provider 线上 A/B
- 复杂 Provider 评测矩阵

### 不做前端侧 Prompt 能力

本阶段明确不做：

- 前端持有 Prompt 正文
- 前端选择 Prompt 版本
- 前端控制 Provider 细节

### 不做复杂实时能力

本阶段不要求：

- `SSE`
- `WebSocket`
- 推送式任务状态流

前端状态跟踪继续遵循 `Polling-First`。

### 不做复杂运营后台与人工流程

本阶段不要求：

- 对比页正式交付
- 多角色后台
- 专用处置工作台
- 大规模运营看板

## 本阶段交付物

### 文档交付

本阶段至少完成：

- Phase 1 范围说明
- 应用层边界说明
- API v0 总契约
- 任务状态机与错误语义
- Schema 版本治理说明
- Prompt 生命周期说明
- Evals 基线说明

### 契约交付

本阶段至少冻结：

- API v0 路由与资源语义
- 联合输入最小提交对象
- 任务状态枚举与流转规则
- 正式输出结构真源位置与版本规则
- 错误对象与错误码最小集合

### 实现准备交付

本阶段应支持后续进入：

- `apps/api/` 最小接口开发
- `packages/application/` 最小用例编排开发
- `packages/schemas/` 正式结构落位准备
- `prompts/` 最小可治理资产落位
- `evals/` 最小样本与基线执行准备

## 与前端协作的边界

本阶段后端应满足前端的最小协作需求：

- 提供稳定的任务创建接口
- 提供稳定的任务状态读取接口
- 提供稳定的结果读取接口
- 保持结果状态与任务状态语义可预测
- 不向前端暴露未经校验的原始模型输出

前端在本阶段继续基于：

- Mock-First
- Adapter-First
- Polling-First

推进页面与适配层开发。

## 与后续阶段的切分原则

### Phase 2 之后再进入的内容

后续阶段再逐步引入：

- 更细粒度的 Prompt 正文资产
- 更细粒度正式 Schema 文件落地与版本化扩展
- API / Worker / 应用层运行时实现
- 更细粒度的阶段评测与报告

### 切分原则

当前阶段优先遵循：

- 先主线稳定
- 再结构细化
- 先接口与结果契约稳定
- 再运行时复杂度增加

## 验收标准

满足以下条件时，可认为 `Phase 1` 范围已冻结：

- 团队可以明确判断某项功能是否属于 `Phase 1`
- API、应用层、Schema、Prompt、Evals 的边界不再互相抢定义权
- 前端可以基于 API v0 与状态语义稳定联调
- 后端可以在不再次澄清范围的前提下开始最小实现
- 后续内容不会被混入本阶段开发
