# Schema 版本治理说明

## 文档目的

本文档定义小说智能打分系统中正式结构契约的版本治理规则，确保 `packages/schemas/` 作为唯一真源时，API、Prompt、Evals 与实现层围绕同一套结构演进。

本文档回答的问题是：

- 什么结构属于 Schema 治理范围
- Schema 如何分类
- Schema 版本应如何命名和升级
- 哪些变更属于兼容变更，哪些属于破坏性变更
- Schema 变更时必须联动哪些资产

本文档不负责：

- 直接定义字段本体
- 直接定义 API 路径
- 直接定义前端 View Model

## 真源原则

正式结构定义以 `packages/schemas/` 为唯一真源。

这意味着：

- `docs/` 中的文档只负责解释结构含义和治理规则
- `apps/api/` 不应自定义第二套正式结构
- `prompts/` 不应成为正式结果结构真源
- 前端最小假契约不应反向替代正式 Schema

## 治理范围

以下结构属于 Schema 治理范围：

- API 请求结构
- API 响应结构
- 正式结果结构
- 中间阶段结构
- Evals 输入输出结构
- 错误对象的正式结构定义

## Schema 分类

建议将 Schema 按用途区分为：

### 1. Request Schema

用于定义请求输入结构。

### 2. Response Schema

用于定义 API 对外响应结构。

### 3. Result Schema

用于定义正式结果对象结构。

### 4. Stage Schema

用于定义后端内部中间阶段结构。

### 5. Eval Schema

用于定义评测输入、输出、报告与比较对象结构。

## 版本标识策略

### 基本原则

每类正式结构都应具备版本标识，并支持在以下层面关联：

- `schemaVersion`
- `promptVersion`
- `rubricVersion`
- `providerId`
- `modelId`

### 推荐规则

推荐采用语义化版本思想：

- 主版本：破坏性变更
- 次版本：兼容扩展
- 修订版本：不改变结构语义的修正

说明：

- 若当前阶段不立即引入完整 SemVer，也应至少明确“破坏性变化必须升级主版本”

## 兼容性定义

### 兼容变更

以下变更通常视为兼容变更：

- 新增可选字段
- 新增不影响旧字段语义的结构扩展
- 补充字段说明或约束描述但不改变结构含义

### 非兼容变更

以下变更视为破坏性变更：

- 删除已有字段
- 重命名已有字段
- 修改已有字段类型
- 修改字段必需性
- 修改已有字段的业务语义
- 调整枚举值含义

## 变更类型矩阵

### 新增字段

- 若为可选字段：通常兼容
- 若为必填字段：通常非兼容

### 删除字段

- 默认非兼容

### 字段改名

- 默认非兼容

### 类型变化

- 默认非兼容

### 枚举扩展

- 需谨慎评估
- 若调用方对未知枚举无处理能力，可能构成破坏性影响

## 变更流程

### 变更前

至少需要明确：

- 变更原因
- 影响范围
- 是否兼容
- 是否需要升级版本

### 变更时

必须同步检查：

- API 文档
- Prompt 生命周期约束
- Evals 基线
- 前端适配层假设契约

### 变更后

必须留下：

- 版本记录
- 影响说明
- 回归范围说明

## Schema 与 API 的关系

- API v0 只定义接口语义与资源边界
- 正式字段结构应由 `packages/schemas/` 维护
- API 文档引用 Schema，而不是复制 Schema 成为第二真源

## Schema 与 Prompt 的关系

- Prompt 输出必须以正式 Schema 为目标
- Prompt 变更不得绕开 Schema 治理
- Prompt 生命周期中必须声明其依赖的 Schema 版本

## Schema 与 Evals 的关系

- Evals 样本与基线比较必须基于稳定 Schema 版本
- Schema 变更后，应明确是否需要重新生成基线
- Evals 报告应记录所使用的 Schema 版本

## 目录与命名建议

以下为建议的正式结构组织方式，不要求当前仓库已全部落位：

- `packages/schemas/input/`
- `packages/schemas/output/`
- `packages/schemas/evals/`
- 后续按阶段扩展内部中间结构

命名要求：

- 语义清晰
- 不混入页面词汇
- 不使用临时实验命名作为正式名称

## 废弃策略

若某一结构版本准备废弃，应至少说明：

- 废弃原因
- 替代版本
- 兼容窗口
- 哪些调用方需要迁移

## 最小治理要求

在 `Phase 1` 中，至少应做到：

- 正式结果结构有明确真源
- API 请求与响应结构不再散落在多个目录中重复定义
- 任何结构变更都知道要同步哪些文档和资产

## 完成标准

满足以下条件时，可认为 Schema 版本治理已具备最小可用性：

- 团队能判断某个结构变化是否属于破坏性变更
- 团队知道结构变更后需要同步哪些资产
- Prompt、API、Evals 不会绕开 Schema 真源自行演进
- 结构文档与正式结构定义之间职责清晰
