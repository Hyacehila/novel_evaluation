# 评分流水线

本文档定义小说智能打分系统的标准评分流程。

当前应区分两个层次：

- **基线评分流程**：当前仓库阶段优先建立的最小正式链路
- **目标评分流程**：后续演进到分层 `rubric` 评分架构后的正式链路

相关专题文档：

- `docs/architecture/layered-rubric-evaluation-architecture.md`
- `docs/contracts/rubric-stage-contracts.md`
- `docs/decisions/ADR-004-layered-rubric-evaluation.md`

## 基线评分流程

1. 接收用户输入
2. 校验输入结构与基本边界
3. 构建评分任务对象
4. 选择 Prompt 版本与评分模板
5. 选择 Provider 与执行策略
6. 调用模型生成评分结果
7. 校验严格 JSON 输出
8. 执行结果修正或失败处理
9. 返回正式结构化结果
10. 记录日志与可选评测信息

## 目标评分流程

在分层 `rubric` 架构下，标准流程应演进为：

1. 接收用户输入
2. 执行输入预检查与可评性判断
3. 构建评分任务对象
4. 选择 `rubric` 版本、Prompt 版本与执行路径
5. 选择 Provider 与执行策略
6. 抽取结构化证据
7. 执行原子子项评分
8. 执行一致性校验与逻辑校准
9. 在必要时触发升级仲裁
10. 生成正式结果对象并校验严格 JSON 输出
11. 返回正式结构化结果
12. 记录日志、阶段元数据与评测信息

## 每一步的职责

### 输入接收

- 只处理系统边界输入
- 不在这里处理深层业务规则

### 输入校验

- 校验文本是否存在
- 校验输入类型是否合法
- 校验用户配置结构是否合法

### 任务构建

- 为后续执行统一上下文
- 记录 Prompt、Schema、Provider 的版本信息

### Prompt 与 Rubric 选择

- 根据评分任务选择合适 Prompt
- 根据输入类型、评分模式和版本信息选择对应 `rubric`
- 不允许前端传入正式评分 Prompt 正文
- Prompt 与 `rubric` 的版本关系应可追踪

### Provider 执行

- 使用统一抽象接口发起模型调用
- 不将具体供应商细节暴露给上层
- 默认优先支持低成本模型承担结构化子任务
- 对低置信度、高冲突或高价值任务保留升级到更强模型的能力

### JSON 校验

- 将输出视为必须满足的结构契约
- 非法输出进入失败路径
- 对外正式结果与对内阶段契约都应是严格 JSON
- 中间阶段契约只作为后端内部治理对象，不直接暴露为前端正式结果

### 失败处理

- 可以记录错误
- 可以触发重试策略
- 但不得返回无法解析的伪结构化结果
- 应区分输入不可评、证据不足、阶段契约失败、一致性冲突、Provider 失败等不同失败语义
- 对需要升级仲裁的情况，应记录触发原因而不是静默重跑

## 设计目标

- 流水线标准化
- 错误路径清晰
- 契约优先
- 支持未来多 Provider 与回归评测
