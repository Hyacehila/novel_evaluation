# 前端技术路线

## 文档定位

本文用于在现有前端业务文档基础上，明确 `apps/web` 的实现基线，使前端在后端尚未完整展开设计前，仍可按统一路线直接开工。

本文回答“前端用什么技术实现、按什么实现策略落地”，但不替代页面规格、状态流、前后端边界或正式 Schema。

## 技术路线结论

当前前端实现基线确定为：

- 框架：`Next.js (App Router) + React + TypeScript`
- 样式与基础 UI：`Tailwind CSS + shadcn/ui`
- 服务端状态与请求缓存：`TanStack Query`
- 表单状态与前端边界校验：`React Hook Form + Zod`
- 页面局部状态：优先使用 React 内建状态与 URL 状态
- 结果可视化：首期建议使用 `Recharts`，仅在结果页按需引入

## 为什么采用这条路线

### 1. 最适合 Workflow-First

当前前端核心不是内容生产器，也不是 Prompt 调试器，而是围绕“新建任务 -> 查看状态 -> 进入结果 -> 历史回访”组织的评测任务工作台。

`Next.js App Router` 天然适合按页面流和层级布局组织入口、导航与页面壳，适合当前已经稳定的页面树与概念路由语义。

### 2. 最适合异步任务与结构化结果展示

当前前端需要重点处理：

- 任务创建
- 异步任务状态跟踪
- 结果页读取
- 历史记录检索
- 异常结果阻断

这类场景本质上以“服务端状态”为主，而不是复杂前端本地状态为主。`TanStack Query` 比引入全局业务状态中心更贴合当前阶段。

### 3. 最适合先结构后实现

当前后端尚未进入完整设计，因此前端必须能在 Mock、最小假契约和适配层的前提下先搭建页面与工程骨架。

`Next.js + TanStack Query + Zod` 的组合有利于：

- 先写页面壳与流程
- 先接 Mock 数据
- 后续再将 API 适配到真实后端
- 将潜在返工收缩在适配层与映射层

### 4. 成熟度与团队协作成本平衡较好

该路线的工程生态、组件生态、表单校验生态和异步数据生态都较成熟，适合作为当前阶段的稳定工程基线。

## 渲染与路由策略

### 路由基线

前端路由基线与当前信息架构一致：

```text
/
/tasks/new
/tasks/{taskId}
/tasks/{taskId}/result
/history
/compare
```

其中：

- `工作台首页`、`新建评测任务页`、`任务详情 / 状态页`、`结果详情页`、`历史记录页` 为首期核心页
- `结果对比页` 保留路由位置，但只进入结构预留状态

### 渲染基线

当前采用“壳层稳定、数据客户端获取”的策略：

- `app/` 路由层主要负责页面入口、布局层级和参数承接
- 页面业务数据优先通过客户端查询获取
- 当前不要求依赖 SSR 才能完成业务主流程
- 当前不以 `Next.js Route Handlers` 作为核心业务 API 承载方式

这样做的原因是：

- 后端主实现将位于 `apps/api/`
- 前端当前更需要稳定页面壳和交互流，而不是绑定服务端渲染细节
- 先用 Mock / 假契约即可开始页面实现

## 状态分层策略

前端状态按以下层级处理：

### 1. 服务端状态

由接口返回、需要缓存、失效、重试或轮询的数据，统一归为服务端状态，包括：

- 工作台首页摘要数据
- 任务详情数据
- 结果详情数据
- 历史记录列表数据

这部分统一交由 `TanStack Query` 管理。

### 2. 表单状态

输入页中的文本、上传文件、校验错误、提交中状态，统一交由 `React Hook Form` 管理，并通过 `Zod` 完成前端边界校验。

### 3. 页面局部状态

只在局部页面内部短期存在的交互状态，例如：

- 筛选面板开关
- 局部折叠展开
- Tab 切换
- 临时高亮

优先使用：

- `useState`
- `useReducer`

### 4. URL 状态

会影响页面可分享性、返回行为或刷新后恢复能力的状态，应放入 URL，例如：

- 历史记录搜索词
- 历史记录状态筛选
- 当前任务标识

## 当前不采用的状态方案

首期不采用：

- `Redux`
- `MobX`
- 以全局 store 复制服务端状态的方案
- 为了“统一状态”而把页面表单状态、查询状态和 UI 状态混进同一个中心仓库

原因是当前前端主复杂度来自任务流与接口状态，而不是复杂前端领域推导。

## 数据获取总策略

### 1. Query / Mutation 分离

- 读取类操作使用 Query
- 创建类操作使用 Mutation
- 页面不直接调用裸 `fetch` 来散落处理业务请求

### 2. Polling-First

首期异步任务状态采用 `Polling-First`：

- 先以轮询承接任务状态跟踪
- 不将 `SSE` / `WebSocket` 作为首期前提
- 后续若后端成熟，再评估升级为推送机制

### 3. 适配层优先

页面不直接消费原始后端 DTO，而是：

- 接收接口返回
- 在适配层映射为前端 View Model
- 页面只消费 View Model

这样即使后端后续调整字段命名或拆分接口，也尽量不把返工扩散到页面层。

## 表单与上传策略

### 输入表单

输入页采用：

- `React Hook Form` 管理输入与提交状态
- `Zod` 管理前端边界内校验
- 统一错误提示层，不在页面各处分散拼接校验逻辑

### 文件上传

前端对文件仅承担边界内责任：

- 文件存在性校验
- 基础格式校验
- 必要大小限制校验
- 上传失败提示

前端不负责：

- 正式内容解析
- 结构提取
- 文本修复
- Prompt 相关处理

## UI 组织策略

### 1. 页面先行，组件后收敛

当前阶段以页面为主线，先完成：

- 页面骨架
- 页面状态
- 页面流转
- 页面展示边界

只有在多个页面重复使用时，才将 UI 抽取为共享组件。

### 2. 共享 UI 只做真正共享的东西

共享 UI 主要包括：

- 按钮
- 输入框
- 表单控件
- 卡片
- Badge
- 对话框
- Skeleton
- 空态/错误态基础壳

不建议在项目开始阶段就构建过重的“组件体系工程”。

### 3. 结果可视化保持辅助层地位

结果页中的图表和可视化只作为辅助层：

- 不替代正式结构化结果
- 不推断后端未返回的数据
- 不为了图表强行改造核心结果字段

## 错误与异常处理策略

### 路由级错误壳

应用壳层应提供统一错误边界，用于承接：

- 页面装载失败
- 路由级异常
- 非预期渲染错误

### 功能级错误态

具体业务页面仍需保留自身的展示态：

- `load_failed`
- `fetch_failed`
- `empty`
- `filtered_empty`
- `blocked`

### 阻断态规则

当结果不满足正式结果展示条件时：

- 不展示正文
- 不伪装为“部分结果页”
- 只展示阻断说明、状态说明与返回入口

## Mock-First 开工策略

在后端尚未完成设计时，前端按如下顺序推进：

1. 先实现路由壳与页面骨架
2. 再实现 View Model
3. 再实现 Mock 数据与最小假契约适配
4. 最后在不改页面结构的前提下接入真实后端

## 当前明确不纳入的内容

本文当前不固化：

- Prompt 前端化能力
- 前端直连模型能力
- 多角色后台管理体系
- 微前端拆分
- 实时推送架构
- 结果对比页详细实现
- 复杂运营分析大盘
