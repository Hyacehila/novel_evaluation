# 分层 Rubric 评分架构

## 目标

本文档定义小说智能打分系统的目标评分架构：以 `LLM as Judge` 为核心，但不将复杂判断全部压到单次大模型直评上，而是通过分层 `rubric`、结构化中间产物和升级仲裁机制，将复杂评价任务拆解为可治理、可评测、可逐步演进的评分流水线。

该架构需要同时满足以下目标：

- 降低系统对单个高成本模型的强依赖
- 保持对外正式结果结构稳定
- 提高评分可解释性与可追踪性
- 为 Prompt 版本治理、Schema 治理与 Evals 回归提供稳定抓手
- 支持后续逐步从单次评分过渡到多阶段评分

说明：

- 本文档定义的是当前正式评分主线的架构约束
- 当前阶段的正式实施应以本文档、阶段契约文档与实施计划文档共同推进

## 适用范围

当前一等公民输入对象：

- 小说开篇
- 章节节选
- 小说大纲

受限输入对象：

- 其它相关文本可进入 `other` 路径
- `other` 路径默认应降低置信度预期，并允许更早进入不可评或升级判断

当前阶段重点：

- 先定义文档、契约和评测边界
- 不直接要求一次性实现完整多阶段运行时
- 不改变前端不得持有正式 Prompt 的治理原则
- 不改变正式输出应以严格 JSON 为唯一返回形态的约束

## 核心原则

### 1. 证据先于结论

任何评分结论都应尽量绑定证据，而不是只输出抽象评价。评分过程中的中间结论应优先沉淀为可追踪的证据对象，再进入后续评分或聚合环节。

### 2. 小模型优先

低成本模型优先承担边界清晰、结构明确、可局部验证的任务，例如：

- 输入预检查
- 显性风险识别
- 文本证据抽取
- 原子子项评分
- 一致性复核

### 3. 升级基于信号，而不是默认发生

更强模型只在以下情况被调用：

- 低置信度
- 关键维度冲突
- 证据不足但样本价值高
- 结构化校验无法通过
- 高价值样本需要更稳妥判断

### 4. 对外契约稳定，对内契约分层

系统对外仍返回统一正式结果对象，但在系统内部允许存在多个阶段契约，用于承接预检查、证据抽取、原子评分、一致性校验、聚合与仲裁等环节。

### 5. 先建立基线，再提高复杂度

当前仓库仍处于结构建设阶段，因此目标架构应允许渐进落地：

- 第一阶段先建立与 `rubric` 对齐的基线评分链路
- 第二阶段再拆出证据抽取与原子评分
- 第三阶段再接入一致性校验与升级仲裁

## Rubric 层次设计

### 一级维度

当前建议将内部 `rubric` 设计为“细粒度内部维度 + 稳定对外结果字段”的双层结构。

| 内部一级维度 | 说明 | 主要影响的对外字段 |
| --- | --- | --- |
| `marketAttraction` | 市场吸引力与签约潜力 | `commercialValue`、`signingProbability` |
| `narrativeExecution` | 叙事执行、信息组织与语言控制 | `writingQuality` |
| `characterMomentum` | 人物欲望、关系张力与情绪驱动 | `commercialValue`、`writingQuality` |
| `noveltyUtility` | 新鲜度、设定可用性与题材差异化 | `innovationScore` |
| `fatalRisk` | AI 味、套路污染、不可评与致命硬伤 | 四个顶层分数的上限与风险标签 |

说明：

- 顶层四个分数仍然保留为正式外部结果字段
- 内部一级维度不必与外部字段一一对应
- `fatalRisk` 更适合作为“约束维度”而不是普通加权项

### 二级子维度建议

#### `marketAttraction`

- `hookStrength`：开场钩子是否能建立读者继续阅读动机
- `conflictContinuity`：冲突是否具有持续推进潜力
- `retentionPotential`：是否具备连载追更驱动力
- `platformFit`：内容调性与平台读者预期是否匹配

#### `narrativeExecution`

- `clarity`：叙述是否清楚、易读、不混乱
- `pacingControl`：节奏是否有效，是否出现拖沓或过快跳跃
- `informationOrganization`：信息投放是否有层次，是否堆叠说明书式信息
- `languageControl`：语言是否稳定，是否存在明显 AI 味、概念堆砌或名词污染

#### `characterMomentum`

- `protagonistDrive`：主角欲望、目标与行动性是否成立
- `characterDifferentiation`：角色是否有区分度与辨识度
- `relationshipTension`：人物关系是否构成张力
- `emotionalTraction`：情绪推进是否足以支撑阅读惯性

#### `noveltyUtility`

- `freshness`：核心卖点是否具有可感知的新鲜度
- `settingUsefulness`：设定是否真正服务叙事，而非堆概念
- `themeControl`：题材混搭是否受控，是否出现风格污染
- `differentiation`：与常见套路相比是否有明确差异点

#### `fatalRisk`

- `aiManualTone`：明显 AI 教程腔、说明书腔、拼装感
- `staleFormula`：老套路低级复制、低质爽点拼贴
- `conceptSpam`：术语轰炸、命名污染、系统面板污染
- `nonNarrativeSubmission`：输入并非可评的小说文本
- `insufficientMaterial`：文本长度、完整性或信息量不足，导致无法可靠评价

## 原子评分锚点

建议原子评分使用内部统一五档制，再映射到外部 `0-100` 区间。

| 档位 | 含义 |
| --- | --- |
| `0` | 不可评、严重失败或直接命中否决条件 |
| `1` | 明显薄弱，存在结构性问题 |
| `2` | 勉强成立，但缺陷显著 |
| `3` | 合格，可作为普通连载文本处理 |
| `4` | 明显突出，具备较强竞争力 |

原子评分必须满足：

- 至少绑定一个证据项
- 必须给出简短理由
- 必须包含置信度
- 如命中 `fatalRisk`，需要给出风险标签

## 分层评分流水线

### 第 1 层：输入预检查层

职责：

- 判断文本是否属于可评范围
- 判断输入类型更接近开篇、章节还是大纲
- 检查长度、完整性、可读性与基本语言异常
- 识别明显非小说文本、垃圾文本或高噪声文本

输出：`InputScreeningResult`

默认模型策略：

- 优先使用小模型或规则检测
- 能使用确定性校验的地方，不依赖 LLM

失败模式：

- 错误地将可评文本判为不可评
- 未识别出非小说文本
- 输入类型误判，导致后续 rubric 选择错误

升级条件：

- 输入类型不明确
- 可评性判断低置信度
- 高价值样本但文本边界模糊

### 第 2 层：证据抽取层

职责：

- 抽取人物、冲突、设定、叙事推进、语言特征等结构化证据
- 抽取可直接支撑评分的显性观察项
- 对同一段文本尽量复用证据，而不是在多个子项中重复生成自然语言说明

典型证据对象：

- 关键人物与目标
- 初始冲突与悬念
- 世界观设定与使用方式
- 节奏异常点
- AI 味或术语污染迹象
- 平台适配线索

输出：`EvidencePack`

默认模型策略：

- 使用小模型完成结构化提取
- 优先生成短证据片段和标签，而不是长篇分析

失败模式：

- 证据抽取过于空泛
- 证据与原文无绑定关系
- 证据漏抽导致后续评分偏低或偏高

升级条件：

- 核心维度缺少足够证据
- 证据彼此明显冲突
- 抽取结果与预检查类型不一致

### 第 3 层：原子评分层

职责：

- 基于 `rubric` 子维度独立评分
- 每个子维度输出分数、理由、证据引用和置信度
- 将“结论必须可追踪到证据”固化为阶段契约

输出：`RubricScoreCard`

默认模型策略：

- 由小模型按子项并行评分
- 对边界清晰的负向信号可结合规则直接给出风险标记

失败模式：

- 多个子项重复使用同一抽象理由但未区分逻辑
- 对没有证据支撑的维度强行打分
- 同时出现高分与强风险但未解释

升级条件：

- 关键维度得分差异异常
- 高分结论缺少对应证据
- `fatalRisk` 与主维度结论冲突

### 第 4 层：一致性校验层

职责：

- 检查维度间是否互相矛盾
- 检查是否存在重复计分、证据不足、空话解释
- 检查原子评分是否与预检查、证据抽取结果一致

输出：`ConsistencyCheckResult`

默认模型策略：

- 小模型复核 + 规则检查混合
- 优先发现冲突，不负责重新生成完整结果

失败模式：

- 放过明显冲突
- 误报冲突导致不必要升级

升级条件：

- 关键冲突数量超过阈值
- 一致性校验无法给出明确结论
- 最终聚合可能受到高影响冲突污染

### 第 5 层：聚合与校准层

职责：

- 将原子评分聚合为一级维度结果
- 将一级维度结果投影为正式外部输出字段
- 生成风险标签、改进重点、置信度和摘要说明
- 执行顶层分数之间的逻辑校准

聚合原则：

- `commercialValue` 主要受 `marketAttraction` 影响，次级参考 `characterMomentum`
- `writingQuality` 主要受 `narrativeExecution` 影响，次级参考 `characterMomentum`
- `innovationScore` 主要受 `noveltyUtility` 影响，并受 `staleFormula`、`conceptSpam` 约束
- `signingProbability` 不应脱离前三项基础分独立飙升，并应受 `fatalRisk` 的上限约束

输出：`AggregatedRubricResult`

失败模式：

- 聚合后顶层分数与下游证据不匹配
- 风险标签与最终结论割裂
- 逻辑校准缺失导致顶层分数相互矛盾

升级条件：

- 校准后仍存在明显矛盾
- 顶层字段置信度过低
- 高价值任务需要更强模型做最终仲裁

### 第 6 层：升级仲裁层

职责：

- 只处理争议项，而不是默认重跑全部流程
- 对低置信度、高冲突、高价值样本进行升级复核
- 输出维持、修正或不可评判定

推荐触发信号：

- `overallConfidence` 低于阈值
- 核心冲突数超过阈值
- 关键字段缺少足够证据支撑
- 评测模式要求更高稳定性
- 人工复核模式要求更保守结论

输出：`ArbitrationRecord`

仲裁要求：

- 必须说明升级原因
- 必须指出争议点
- 必须记录维持或修正的依据
- 不允许将仲裁过程变成新的黑盒整体重写

## 默认低成本路径与高风险路径

### 默认低成本路径

`输入预检查 → 证据抽取 → 原子评分 → 一致性校验 → 聚合与校准 → 正式结果`

适用：

- 普通用户样本
- 证据较充分的主流文本
- 对时延和成本较敏感的在线评分场景

### 高风险升级路径

`输入预检查 → 证据抽取 → 原子评分 → 一致性校验 → 聚合与校准 → 升级仲裁 → 正式结果`

适用：

- 低置信度样本
- 边界类型样本
- 高价值样本
- 回归基线验证样本

## Rubric 治理真源

当项目进入分层 `rubric` 实施阶段后，`rubric` 将成为独立治理对象，因此应明确唯一真源规则：

- 正式 `rubric` 资产应放在 `prompts/scoring/rubric/`，它是真源于评分规则本体
- `docs/` 中的 `rubric` 文档负责解释架构、边界与字段语义
- `packages/schemas/` 负责定义与 `rubric` 对应的正式结构，它是真源于结构定义
- 不允许在多个位置分别维护互相独立的正式 `rubric` 本体

版本治理建议：

- `rubricVersion` 应作为正式版本字段存在
- `rubric` 变更必须同时检查 Prompt、Schema 与 Evals
- 维度标识符变更应视为高影响变更，并记录变更原因

## Prompt 与 Schema 映射

### Prompt 建议分层

- `prompts/scoring/system/`：系统级评分宪法 Prompt
- `prompts/scoring/rubric/`：Rubric 维度与评分口径 Prompt
- `prompts/extraction/`：证据抽取 Prompt
- `prompts/calibration/`：一致性校验、校准与仲裁 Prompt

### Schema 建议分层

后续正式结构建议沉淀到：

- `packages/schemas/input/`：输入预检查相关结构
- `packages/schemas/output/`：正式结果结构与阶段聚合结构
- `packages/schemas/evals/`：评测样本、报告与阶段比较结构

说明：

- 正式对外结果仍以最终输出 Schema 为唯一真源
- 中间阶段结构应作为内部契约维护，不应泄露给前端作为正式业务接口

## 与现有输出契约的关系

当前系统对外仍保留以下顶层评分字段：

- `signingProbability`
- `commercialValue`
- `writingQuality`
- `innovationScore`

分层 `rubric` 机制的目标不是替换这些字段，而是：

- 提高这些字段的生成过程可解释性
- 提高这些字段在 Prompt、Provider 与版本变化下的稳定性
- 为 `strengths`、`weaknesses`、`editorVerdict`、`marketFit`、`detailedAnalysis.*` 提供更一致的证据来源

## 渐进落地建议

### Phase 1：Rubric 对齐基线

- 冻结一级维度与子维度词表
- 建立阶段契约文档
- 保持轻量基线评分链路，但要求结果解释与 `rubric` 对齐

### Phase 2：证据抽取与原子评分

- 将证据抽取与原子评分拆分为独立 Prompt 与独立中间契约
- 评估小模型在结构化子任务上的稳定性

### Phase 3：一致性校验与升级仲裁

- 增加冲突检测、置信度路由与升级仲裁
- 在 Evals 中比较“基线路径”与“增强分层路径”的成本、稳定性与质量差异

## 成功标准

- 系统能解释每个顶层评分主要来自哪些内部维度和证据
- 默认路径能由低成本模型承担大部分结构化子任务
- 更强模型只在必要时升级调用
- Prompt、Schema 与 Evals 可以围绕同一套 `rubric` 版本同步演进
- 对外正式结果结构保持稳定，不因内部架构演进而频繁变更
